# Ask Your Bookshelf — Stage 1, Plan 2: The OKF Engine — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the Plan 1 stub `ContextSourceStrategy` with a real OKF engine that compiles the user's whole library into a cross-linked markdown knowledge base once (cached), then answers "ask your bookshelf" questions by progressive disclosure over that base — drawing only from books the user can access, with source-book citations. After Plan 2, shelf discussions answer from the actual 28 books instead of apologizing.

**Architecture:** Compile-once wiki (per-book JSON concept extraction → topic cluster → per-cluster cross-book themes → rendered markdown under `STORAGE_PATH/shelf-wiki/`); query-time progressive disclosure (read root `index.md` → LLM picks cluster(s) → LLM picks concept file(s) → answer with citations). The LLM emits JSON throughout; the script renders markdown → dangling links are impossible by construction. The engine implements `ContextSourceStrategy`; `getContextSource()` swaps the stub for `OkfContextSource`. OpenRouter-only (uses `getShelfLlmConfig` = admin key + optional model override); `completeChat({jsonMode:true})` for all structured calls. No new external deps.

**Tech Stack:** TypeScript, Prisma/SQLite (read-only for the engine — no schema change), the `storage` adapter (`src/server/storage/local.ts`), `chunkText` (`src/lib/tts/chunk.ts`), `completeChat` (`src/server/services/openrouter.ts:291`), Vitest.

## Global Constraints

- **No schema changes** (Plan 1's nullable `bookId` already covers this plan).
- **No new dependencies.** OpenRouter via the existing `completeChat` + `getShelfLlmConfig` (admin key). Embeddings are Stage 2 (RAG), NOT this plan.
- **The LLM never writes markdown.** Extraction/synthesis return JSON; `render.ts` writes files. Links are generated by the script between concept IDs that exist.
- **Caching is mandatory:** every LLM JSON output keyed by content hash under `STORAGE_PATH/shelf-wiki/.cache/` → re-runs are incremental/free; adding one book costs only that book + its cluster.
- **Access control:** query-time retrieval filters concepts to the user's `getAccessibleBookIds(userId)` set. Compile is global over all `EpubFile` rows.
- **`completeChat` contract:** `completeChat({apiKey, model, prompt, systemMessage?, jsonMode:true})` returns a `Promise<string>`; the caller `JSON.parse`es. The engine wraps it in `llm-json.ts` (parse + shape-validate + retry-once).
- **Run tests:** `npm test`. **Lint:** `npm run lint` (322-error baseline pre-existing — no NEW errors).
- **Cost gate:** the full 28-book compile (Task 11) is the only step that spends real money. `build-wiki.ts preview()` reports token/chunk/call estimates + the configured model BEFORE any spend, and the controller (you) approves before running the real compile. Re-runs are cached/free.
- **Ponytail house style:** `// ponytail:` comments mark deliberate simplifications (name the ceiling + upgrade path). No other comments unless asked.
- **Path sanity:** the repo is at `/Volumes/Crucial-6TB/Dev/nutshell-via-pi` (no trailing "a"). Double-check before any write.

### Task 3: `render.ts` — JSON concept-data → markdown (the anti-dangling-link core)

**Files:** Create `render.ts`; test `__tests__/render.test.ts`
**Interfaces:** Consumes `OkfConcept` (+ a `OkfClusterTheme` type added to `types.ts`: `{ topic, title, summary, relatedConceptIds: string[] }`). Produces:
- `conceptToMarkdown(c) → { relPath, body }` where `relPath = concepts/<sourceBookId>/<slug(title)>.md`, frontmatter `{type, title, sourceBookId, topic}`, body from `bodyFields`.
- `themeToMarkdown(t, resolvedLinks) → { relPath, body }` — links rendered ONLY to concept IDs in the passed `resolvedLinks` set (this is the dangling-link impossibility: the renderer never emits a link the caller hasn't validated).
- `buildIndex(wiki) → { relPath, body }` — hierarchical `index.md` (clusters → books → concepts) with one-line descriptions, link targets the renderer KNOWS exist.
- `slug(s)` — lowercase, alphanum+hyphen, deterministic.

- [ ] Steps 1-5: TDD — `slug` deterministic; `conceptToMarkdown` produces valid frontmatter + relPath; `themeToMarkdown` OMITS links not in `resolvedLinks` (the key safety test — pass a theme referencing a missing concept, assert no link to it appears, or it degrades to plain text); `buildIndex` only links known IDs.
- [ ] Commit `feat(shelf-knowledge): render JSON concepts → markdown (dangling-link-safe)`.

### Task 6: `extract.ts` — per-book concept extraction

**Files:** Create `extract.ts`; test `__tests__/extract.test.ts`
**Interfaces:** Produces `extractBookConcepts(book) → Promise<{ concepts: OkfConcept[], topic: string, form: "narrative"|"nonfiction"|"unknown" }>` where `book` is a Prisma `EpubFile` (+ its `bookMetadata`). Pipeline: load `txtPath` text; chunk via `chunkText(text, {softLimit: 6000, hardLimit: 8000})`; for each chunk, `completeJson` with an extraction prompt whose template branches on `bookMetadata.isNarrative` (narrative → characters/themes/settings; nonfiction → arguments/frameworks/evidence/key concepts; unknown → infer `form` and tag it, so the 6 unclassified books get backfilled); merge/dedupe concepts per book (by slug(title)); cache each chunk's JSON by content hash (so re-runs are free). The topic is derived from the extracted concepts (most common / an LLM-assigned topic field per book).

- [ ] Steps 1-5: TDD with `completeJson` mocked — assert: narrative branch uses the narrative prompt; chunk-and-merge dedupes by slug; cache is consulted (a repeat call doesn't re-invoke `completeJson` for the same chunk); unknown-metadata book sets `form` from the model's output.

### Task 9: `query.ts` — progressive disclosure

**Files:** Create `query.ts`; test `__tests__/query.test.ts`
**Interfaces:** Produces `answerShelfQuestion(args: { question, accessibleBookIds }) → Promise<{ prompt, sourceText, bookText, bookMd5, promptVersion, citations: { bookId, bookTitle, conceptTitle }[], debug? }>` (the fields `ContextSourceStrategy.buildContext` needs). Pipeline (2 LLM calls, both cached keyed by question+access-set):
1. Navigate: read root `index.md` (filtered to accessible books' clusters/concepts — build a filtered index string); `completeJson` asks the model to return `{ clusterIds: string[], conceptQueries: string[] }` — which clusters + which concept titles to read.
2. Read + answer: load the selected concept files (filtered by access), `completeJson`/answer with those as context; return the answer text + the citations (concept files carry `sourceBookId` → resolve titles).
If navigation selects nothing useful, fall back to "I couldn't find relevant concepts in your library for that question."

- [ ] Steps 1-5: TDD with `completeJson` + `wiki-storage` mocked — navigate returns ids; query loads those files; answer uses only accessible concepts (a concept whose sourceBookId isn't in `accessibleBookIds` is filtered out before the answer call — assert it's never read); citations resolve to titles.
- [ ] Commit `feat(shelf-knowledge): progressive-disclosure query + access filter`.

### Task 10: `OkfContextSource` — wire the real engine behind the strategy

**Files:** Create `okf-context-source.ts`; modify `context-source.ts`; test `__tests__/okf-context-source.test.ts`
**Interfaces:** `OkfContextSource implements ContextSourceStrategy`: `isReady()` = `wikiExists("concepts/")` (the wiki has been built); `buildContext({question, userId, accessibleBookIds})` = call `answerShelfQuestion`, map the result to `BuiltPrompt` shape (`prompt` = the answer system message + answer text + a citation list; `sourceText`/`bookText` = the navigated concept contents; `bookMd5` = `shelf:${userId}`; `promptVersion` = 2). Handles not-built (`isReady` false → the service already turns this into an error event) and no-relevant-concepts (the query fallback). `getContextSource()` returns `new OkfContextSource()`.
