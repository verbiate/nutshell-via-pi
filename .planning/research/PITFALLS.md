# Pitfalls Research

**Domain:** AI-powered ebook reader with EPUB ingestion, AI explainers, TTS audio, and multilingual support
**Researched:** 2026-05-06
**Confidence:** HIGH

---

## Critical Pitfalls

### Pitfall 1: AI Hallucination Presented as Book Truth

**What goes wrong:**
The AI Explainer invents plot points, misattributes quotes, fabricates character motivations, or confidently explains something that doesn't exist in the book. Users trust it because the UI presents it as "the explanation for this book," creating misinformation at scale.

**Why it happens:**
LLMs are optimized for fluency, not factual accuracy relative to a specific text. Without grounding constraints, they hallucinate. The "Explainer" branding (vs "summary") actually increases trust because it sounds authoritative. OpenRouter's cheapest models (used for Regular tier) hallucinate more than premium ones.

**How to avoid:**
- **Ground every explainer in the source text**: Include relevant passages from the TXT conversion in the LLM prompt as context, with explicit instructions to only reference provided text
- **Add a "confidence" indicator** in the UI: "This explainer is generated by AI and may contain inaccuracies. Always verify against the original text."
- **Use retrieval-style prompting**: For section-level explainers, retrieve the section text + surrounding context and instruct the model to cite specific paragraphs
- **Tiered model quality matters**: Regular tier gets faster/cheaper models with more guardrails; Pro tier gets reasoning models better at faithful summarization
- **Never cache without source text linkage**: Store the prompt context (section text used) alongside the explainer so hallucinations can be audited

**Warning signs:**
- User reports: "The explainer says X happened, but that's not in the book"
- Explainers are suspiciously generic (hallucination mode: vague) or suspiciously specific about things not in the text
- Section explainers for different sections sound structurally identical (model is ignoring the input)

**Phase to address:**
Phase 3 (AI Explainers) — prompt engineering and grounding architecture must be designed before the first explainer is cached; retrofitting is nearly impossible without regenerating everything

### Pitfall 2: TTS Cost Explosion via Cache Misses

**What goes wrong:**
Audio generation costs spiral out of control. A single long book (500+ pages) can cost $10-50 in TTS API fees. Without aggressive caching, every new user who opens a book triggers new audio generation. The caching key is wrong, or the cache is never hit because of language/model/version variations.

**Why it happens:**
TTS is priced per character (ElevenLabs) or per request (fal.ai). Book-level audio for a 300-page novel is ~500,000 characters = ~$5-10 at ElevenLabs rates. If the cache key includes user ID instead of content hash, or if TTS model versions change, the cache is useless. Teams also forget that section-level audio ("explain this chapter") multiplies costs by the number of chapters.

**How to avoid:**
- **Cache key = (content_hash, language, voice_id, tts_model_version)** — never include user ID
- **Pre-compute book-level audio only** for v1; section-level audio is on-demand and cached after first generation
- **Set hard cost limits** per book/audio job in the worker queue: if estimated cost > $X, require admin approval or downgrade to cheaper voice
- **Store audio in cheap object storage** (Cloudflare R2, S3) with CDN — never in the database
- **Track generation cost per book** in the database and surface it in the Admin panel
- **Use TXT length to estimate cost before generating** and warn: "This book will cost approximately $X to generate audio"

**Warning signs:**
- Monthly API bill is 10x higher than projected
- Same book showing audio generation jobs in the queue multiple times
- Users reporting slow audio generation for books that should already have it
- Cache hit rate < 80% (should be > 95% for popular books)

**Phase to address:**
Phase 1 (Core Infrastructure) — caching architecture and cost estimation must be in place before TTS integration; Phase 5 (TTS) for model/version pinning

### Pitfall 3: EPUB Parsing Edge Cases That Corrupt the Reader

**What goes wrong:**
EPUBs with nested TOCs, malformed HTML, external image references, embedded fonts, or DRM remnants break the parser. The reader displays garbled text, missing chapters, or crashes entirely. Some EPUBs are basically ZIPs of XHTML files that reference resources via relative paths that break during TXT conversion.

**Why it happens:**
EPUB is a container format, not a content format. "Valid EPUB" is a loose standard. Self-published EPUBs often have broken XML, missing manifest entries, or HTML so malformed it crashes parsers. The EPUB-to-TXT conversion pipeline is usually the weakest link.

**How to avoid:**
- **Use a battle-tested EPUB parser** (`@likecoin/epub-ts`) rather than hand-rolling ZIP + XML extraction
- **Parse defensively**: wrap every extraction step in try/catch, log failures, and gracefully degrade (skip malformed chapters rather than crash)
- **Validate EPUB on upload**: check ZIP structure, required files (`mimetype`, `META-INF/container.xml`), and fail fast with a clear user message
- **Store the original EPUB** alongside the TXT so the reader can fall back to rendering the original if the TXT is corrupt
- **Test with real-world EPUBs**: Project Gutenberg (clean), modern publishers (complex CSS), and Calibre outputs (varied)
- **Handle encodings explicitly**: force UTF-8 during TXT conversion; EPUBs often declare one encoding but use another

**Warning signs:**
- User uploads trigger 500 errors with stack traces in EPUB parser
- TXT files with `�` replacement characters (encoding issues)
- Missing chapters in the reader that exist in other ebook apps
- TOC shows entries that lead to empty pages

**Phase to address:**
Phase 1 (Core Infrastructure) — parser robustness must be proven before user-facing upload; Phase 2 (Reader) for rendering edge cases

### Pitfall 4: Multilingual TTS Generates Wrong-Language Audio

**What goes wrong:**
A user sets language to Vietnamese, uploads an English book, and requests audio. The TTS model attempts to read English text with Vietnamese phonemes, producing gibberish. Or: the system auto-detects the book's language incorrectly and caches audio in the wrong language.

**Why it happens:**
TTS models are language-specific. ElevenLabs voices are tied to languages; fal.ai has language codes. If the book language != user preference != TTS voice language, the output is garbage. Auto-detection libraries (`franc`, `langdetect`) fail on short texts, mixed-language books, or technical content.

**How to avoid:**
- **Explicit language selection at upload**: detect and suggest, but let the user confirm or override
- **Store `book_language` separately from `user_preference_language`** — the former drives TTS voice selection, the latter drives explainer generation
- **Validate TTS voice compatibility before generation**: if the book is English but the user wants Vietnamese audio, either (a) refuse with explanation, (b) translate first (out of scope for v1), or (c) use an English voice and explain the mismatch
- **Cache by `(content_hash, book_language, voice_id)`** not user preference — prevents wrong-language cache pollution
- **Test language detection on sample corpus**: Project Gutenberg books in 10+ languages, technical texts, mixed-language content

**Warning signs:**
- Audio sounds like nonsense syllables (wrong phoneme set)
- Cache contains "English book, Vietnamese audio" entries that are useless
- Users reporting audio "sounds weird" or "is in a different language"
- Language detection flips between chapters in the same book

**Phase to address:**
Phase 1 (Core Infrastructure) — language metadata model must be designed before first upload; Phase 5 (TTS) for voice-language mapping

### Pitfall 5: Admin Panel as Privilege Escalation Vector

**What goes wrong:**
The admin panel has weak access controls, lacks audit logging, or exposes dangerous operations (delete user, change roles, delete Universal Library book) without confirmation. A compromised Regular user account or a CSRF attack escalates to full admin access.

**Why it happens:**
Admin panels are often built as an afterthought with a simple `if (user.role === 'admin')` check. They're rarely penetration-tested. The "admin-managed roles" requirement means the admin panel is the primary attack surface for privilege escalation.

**How to avoid:**
- **Admin routes are a separate API namespace** (`/api/admin/*`) with middleware that validates role at every layer, not just the UI
- **Audit log every admin action**: who, what, when, old value, new value — stored in an append-only table
- **Require re-authentication for sensitive actions**: changing another user's role or deleting a Universal Library book should require the admin's password again
- **No client-side role checks**: all role validation happens server-side; the UI merely hides buttons the user can't use
- **Rate-limit admin endpoints** separately from user endpoints
- **Default deny**: new features are invisible to non-admins until explicitly allow-listed

**Warning signs:**
- Admin API endpoints return data for non-admin authenticated users (test with a Regular user's JWT)
- Admin actions have no traceability ("who deleted this book?")
- Client-side `display: none` is the only thing hiding admin controls
- Admin endpoints use the same rate limits as public endpoints

**Phase to address:**
Phase 1 (Core Infrastructure) — auth architecture including role-based middleware; Phase 6 (Admin Panel) for UI and audit logging

### Pitfall 6: Large EPUBs Crash the Server

**What goes wrong:**
A user uploads a 200MB EPUB (illustrated textbook, comic, or omnibus edition). The server attempts to read the entire file into memory for MD5 hashing, TXT conversion, or EPUB parsing, causing an out-of-memory crash. Or the TXT file is so large that database queries timeout.

**Why it happens:**
Node.js has a default heap limit (~1.4GB on 64-bit). Reading a 200MB EPUB into a Buffer, then extracting and concatenating all chapters into a single TXT string can easily exceed this. Prisma + SQLite also struggle with multi-megabyte text fields in queries.

**How to avoid:**
- **Stream MD5 computation**: use a streaming hash (Node.js `crypto.createHash` with streams) rather than `fs.readFileSync`
- **Streaming EPUB extraction**: process chapters one at a time, write TXT incrementally to disk, then stream into storage
- **File size limits**: reject EPUBs > 50MB for v1 with a clear message; "Large file support" can be a v2 feature
- **TXT chunking**: store book text as chunked records (e.g., per-chapter) rather than one massive text field; the reader assembles them
- **Database text limits**: SQLite handles large text, but don't SELECT the full TXT in list views — only fetch it when opening the reader
- **Worker queue for heavy jobs**: EPUB processing happens in a background worker, not the request thread

**Warning signs:**
- Server crashes with `FATAL ERROR: Reached heap limit Allocation failed`
- Upload requests timeout for large files
- Database queries for "My Library" are slow because they accidentally JOIN with the full text content
- Memory usage spikes during EPUB processing

**Phase to address:**
Phase 1 (Core Infrastructure) — streaming architecture and size limits must be in place before any user upload

### Pitfall 7: The "Explainer" Brand Dilution

**What goes wrong:**
Despite the explicit requirement to "never call Explainers summaries," the UI, codebase, API endpoints, and database columns all use the word "summary." The product loses its differentiation and users perceive it as just another AI summary tool. The team's own language drifts.

**Why it happens:**
"Summary" is the default mental model. Engineers, designers, and even founders will unconsciously use it in code (`getSummary()`, `summaryText`), documentation, and user-facing copy. It takes active discipline to maintain the distinction.

**How to avoid:**
- **Establish a glossary in code reviews**: any PR using "summary" when "explainer" is meant gets flagged
- **Name database tables and API endpoints explainer-first**: `explainers`, `generateExplainer()`, `/api/explainers`
- **Lint for the word "summary"** in user-facing strings (can be a simple grep in CI)
- **Train the team**: an explainer teaches *why* and *how* something matters, not just *what* happened — this affects prompt engineering too
- **Design review check**: every mock is checked for "summary" language

**Warning signs:**
- API docs reference `/api/summaries`
- Users say "I love the summaries" in feedback and the team doesn't correct them
- Marketing copy accidentally describes "AI summaries of your books"
- The word "summary" appears 50+ times in the codebase

**Phase to address:**
Phase 0 (Project Setup) — glossary and naming conventions; all phases ongoing via code review discipline

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
| --- | --- | --- | --- |
| Store full TXT in a single DB column | Simple schema, easy to query | Queries slow down; memory pressure; hard to paginate | Never for v1 — chunk from day one |
| Use `fs.readFileSync` for EPUB hashing | One line of code | Crashes on files > heap limit; blocks event loop | Never — always stream |
| Skip EPUB validation on upload | Faster upload UX | Corrupt data in Universal Library; hard to debug later | Never — validate before accepting |
| Cache explainer without storing prompt context | Smaller cache records | Cannot audit or regenerate when model changes | MVP only — add context storage before Pro tier launch |
| Single TTS provider (only ElevenLabs) | Faster integration | No failover; no cost optimization; locked in | v1 acceptable if fal.ai integration is Phase 5 |
| Client-side role gating | Faster admin panel build | Bypassable; security theater | Never — all auth server-side |
| SQLite for everything | Zero infra setup | WAL mode locks under concurrent TTS workers; backup complexity | v1 only — plan Postgres migration for v2 if scale demands |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
| --- | --- | --- |
| **OpenRouter** | Not handling model downtime/fallbacks; hardcoding one model ID | Maintain a model priority list per tier; implement automatic fallback with exponential backoff |
| **ElevenLabs TTS** | Sending full book text in one request; not handling rate limits | Chunk text into ~5,000 char segments; queue chunks; retry on 429 with backoff |
| **fal.ai TTS** | Ignoring webhook/async response pattern; polling indefinitely | Implement webhook handler for completion; set max polling time (e.g., 5 min) with dead-letter queue |
| **OpenRouter (prompting)** | Sending entire book text as context for a section explainer | Retrieve only the relevant section + ~2 paragraphs of context; never send full book |
| **Prisma + SQLite** | Using `prisma.$transaction` for long-running audio generation jobs | SQLite WAL mode has concurrent writer limits; use a job queue (Bull, Inngest) for TTS/explainer generation |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
| --- | --- | --- | --- |
| Loading full book text on My Library page | Library list takes 5+ seconds to load | SELECT only metadata (title, author, cover, progress); never JOIN with text content | > 20 books in library |
| Synchronous EPUB processing on upload | Upload request times out at 30s; user sees spinner forever | Return 202 Accepted immediately; process in background worker; notify via WebSocket or polling | Any EPUB > 5MB |
| N+1 queries for book list + access checks | My Library page makes 1 query per book to check access | Use a single query with `WHERE user_id = X` via the access join table | > 10 books in library |
| Storing audio in database BLOBs | DB backups become huge; audio delivery is slow | Store audio files in object storage (R2/S3); DB stores URL + metadata | First audio file > 1MB |
| Rendering entire book in DOM at once | Reader tab crashes on large books; scroll performance terrible | Virtual scroll or chapter-by-chapter rendering; never mount 500k+ words in DOM | > 300-page book |
| No index on `epub_hash` lookups | Universal Library lookup is slow | Index `epub_hash` in `epub_files` table; it's the primary dedup key | > 1,000 books in library |

## Security Mistakes

| Mistake | Risk | Prevention |
| --- | --- | --- |
| Admin panel accessible by guessing the URL | Unauthorized role changes, data deletion | Admin routes under `/admin` require authentication + role check; don't rely on "security through obscurity" |
| EPUB contains malicious HTML/JS | XSS when rendering EPUB content in browser | Sanitize all EPUB HTML before rendering; use DOMPurify; run reader in a separate iframe or CSP-restricted context |
| User ID in cache key leaks across users | User A sees User B's explainer or audio | Cache keys must never contain user ID; use content hash only |
| MD5 collision attack on upload | Attacker uploads malicious file with hash collision of existing book | Extremely unlikely for MD5 with random data, but accept: validate file is valid EPUB before hashing matters |

## UX Pitfalls

| Pitfall | What Goes Wrong | Prevention |
| --- | --- | --- |
| No progress indication during upload | User clicks upload, sees nothing for 30s, clicks again | Show upload progress bar; show "Processing..." state with estimated time; disable re-upload during processing |
| Explainer appears instantly on first request | User expects AI generation to take time; instant result seems fake | Show "Generating your explainer..." for minimum 1s even if cached; distinguish cached vs generated in UI |
| No "empty state" for new users | First login shows blank library; user doesn't know what to do | Onboarding: "Upload your first book" CTA with drag-and-drop zone; sample book option |
| Reader themes don't persist | User sets sepia, refreshes, gets light theme | Persist theme in user profile (database) + localStorage fallback; load before first render to prevent flash |
| "My Library" doesn't show progress | Books all look the same; no sense of accomplishment | Show reading progress % on book covers; "Currently reading" section at top |

## "Looks Done But Isn't" Checklist

Before marking any phase complete, verify:

- [ ] **Auth**: Can a Regular user access admin endpoints? (Should fail with 403)
- [ ] **Upload**: Does a 60MB EPUB get rejected gracefully? (Should fail with clear message)
- [ ] **Dedup**: Upload same EPUB twice — second upload should grant access, not store duplicate
- [ ] **Cache**: Request same explainer twice — second should be instant (cache hit)
- [ ] **Language**: Request explainer in Spanish, then French — should generate two different outputs
- [ ] **Resume**: Close reader mid-chapter, reopen — should return to exact paragraph
- [ ] **Admin**: Change user's role from Regular to Pro — should take effect immediately on next request
- [ ] **TTS**: Request audio for same book twice — second should play immediately from cache
- [ ] **Explainer naming**: Search codebase for "summary" — should return 0 results in user-facing code
- [ ] **Position tracking**: Change font size in reader — should stay at same content position, not scroll percentage
